# Hubitat `zigbee` helper — reference

Reverse-engineered reference for the `zigbee` object Hubitat injects into drivers. Hubitat's [own page](https://docs2.hubitat.com/en/developer/zigbee-object) lists method names without signatures and references `ZigbeeCluster` / `ZoneStatus` classes that aren't documented. This doc fills the gaps by combining live Groovy reflection on a running hub with the original SmartThings docs (Hubitat's helper is descended from SmartThings').

**Sources**
- Reflective dump: `hubitat-zigbee-introspection.md` (in this folder), generated by `drivers/tests/ZigbeeIntrospect.groovy` against firmware 2.5.0.148 on C-8 Pro
- SmartThings docs (Wayback): [zigbee-ref @ 2020-05-10](https://web.archive.org/web/20200510195834/https://docs.smartthings.com/en/latest/ref-docs/zigbee-ref.html), [zigbee-primer @ 2020-10-08](https://web.archive.org/web/20201008180753/https://docs.smartthings.com/en/latest/device-type-developers-guide/zigbee-primer.html), [building-zigbee-DHs @ 2020-09-21](https://web.archive.org/web/20200921015021/https://docs.smartthings.com/en/latest/device-type-developers-guide/building-zigbee-device-handlers.html)
- Code examples (still live): the various `zigbee-*-handler.src/*.groovy` under [SmartThingsCommunity/SmartThingsPublic](https://github.com/SmartThingsCommunity/SmartThingsPublic/tree/master/devicetypes/smartthings)

## How to inspect it yourself

Three sandbox-safe techniques, picked by what you want:

**Just the runtime class name** — `getObjectClassName(value)` (platform-injected, returns the FQN as a String). Documented in `hubitat-platform-notes.md` as the sandbox-safe replacement for `value.getClass().getName()`.

**Live bean-property snapshot** — `obj.properties.each { k, v -> … }`. Returns the Groovy bean-accessor map of all readable getter properties with their current values. The one-liner approach used by `drivers/tests/DeviceInspector.groovy:178` to dump the full `DeviceWrapper` surface. For the `zigbee` object specifically this yields only 6 entries (`class`, `cluster`, `device`, `deviceNetworkId`, `endpointId`, `zigbeeId`) — most `zigbee.*` members take arguments and aren't bean properties — but for `device`, `location`, `hub`, etc. it's the right tool: nearly all of their surface IS bean properties with useful live values.

**Full API surface with signatures** — Class reflection. Needed when you want overloaded method signatures, parameter types, static fields (the cluster constants), or to walk referenced types. The sandbox blocks several routes:
- `obj.getClass()` is rejected as a `MethodCallExpression` → use `obj.class` (PropertyExpression)
- `java.lang.Class` is rejected as an explicit import, which kills `Class cls` parameter annotations → use `def cls` everywhere
- `Class.forName(String)` is rejected at AST level → reach related classes via factory methods (e.g. `zigbee.parseZoneStatus("zone status 0x0000").class`)
- `java.lang.reflect.Modifier.isStatic(...)` is rejected → bitmask directly: `(f.modifiers & 0x08) != 0` for STATIC
- `ArrayDeque` import is rejected → plain `[]` List with `.remove(0)` works as a FIFO

Then walk `cls.getMethods()` / `cls.getFields()` / `cls.getSuperclass()` / `cls.getInterfaces()` and write the result via `uploadHubFile`. The driver in `drivers/tests/ZigbeeIntrospect.groovy` does all three — install on any virtual device, click `dumpAll`, download `zigbee_introspect.md`.

## What the `zigbee` object actually is

- FQN: `com.hubitat.zigbee.Zigbee`
- Extends `java.lang.Object`, implements `groovy.lang.GroovyObject`
- Holds a `DeviceWrapper` reference (`getDevice()` / `setDevice(DeviceWrapper)`), so all the convenience builders know which endpoint to address without you passing it

## Methods (curated, with semantics)

Signatures below are from the live reflection. Semantics in italics are quoted from the SmartThings docs unless noted.

### Parse helpers

| Method | Returns | Notes |
|---|---|---|
| `getEvent(String description)` | `Map` | *"Tries to parse ZigBee Clusters into a map whose key/value pairs can be directly handled by `sendEvent()`."* Only clusters `0x0006` (switch), `0x0008` (level), `0x0702` and `0x0B04` (power), and `0x0300` (color temperature only) are recognized; everything else returns null. Frozen in classic ST; Hubitat hasn't visibly extended it. |
| `parseDescriptionAsMap(String)` | `Map` | The workhorse. Returns at minimum `cluster`/`attrId` (hex strings), `clusterInt`/`attrInt`/`commandInt` (decoded), `value` (hex payload), `isValidForDataType`, and `additionalAttrs` (a list of `[attrInt, value, isValidForDataType]` for multi-attribute reports). Schema undocumented on both sides — verified from real DTHs. |
| `parseZoneStatus(String description)` | `ZoneStatus` | *"The description should be of the form `'zone status {number}'` where `{number}` is a hex number."* Returns `hubitat.zigbee.clusters.iaszone.ZoneStatus` — see below. |
| `parseZoneStatusChange(String)` | `Map` | Hubitat-only (not in ST). Likely parses Zone Status Change Notification (cluster `0x0500` command `0x00`) — confirm with a real IAS sensor before relying. |
| `parse(String)` | `SmartShield` | Low-level catch-all. Returns `hubitat.zigbee.SmartShield`, the raw ZCL frame wrapper (clusterId, command, profileId, source/dest endpoints, data list, manufacturer flags). Most drivers should use `parseDescriptionAsMap` instead. |
| `getKnownDescription(String)` | `Map` | Undocumented on both sides. |

### Low-level cluster I/O

These all take an optional trailing `Map additionalParams = [:]` with keys `mfgCode` (Integer) and `destEndpoint` (Integer) — *"intended to support future params without affecting backward compatibility"*, which explains the map rather than positional args. The trailing `int` overloads carry a delay (ms) for batching.

| Method | Notes |
|---|---|
| `command(Integer cluster, Integer command, String... payload)` | *"Each argument should be passed as an ASCII hex string in little-endian format of the appropriate width for the data type. For example, to pass the value 5 for a UINT24 you would pass `050000`."* |
| `command(Integer cluster, Integer command, Map params, String... payload)` | Same with mfg/endpoint override. |
| `readAttribute(Integer cluster, Integer attributeId)` | Returns `List<String>` of `he raw` commands. |
| `readAttribute(Integer cluster, List attributeIds)` | Multi-attribute read in one frame. |
| `writeAttribute(Integer cluster, Integer attrId, Integer dataType, value)` | ST docs caveat: *"The Integer value to write for data types of boolean, unsigned int, signed int, general data, and enumerations. Other data types are not currently supported but will be added in the future."* Hubitat now overloads with `String` for the value too, so string-class types are reachable. |
| `configureReporting(Integer cluster, Integer attrId, Integer dataType, Integer minSec, Integer maxSec[, Integer reportableChange])` | *"reportableChange required for analog data types. Discrete data types should always provide null for this value."* `minSec` = minimum reporting interval, `maxSec` = max if nothing changes (a heartbeat), `reportableChange` = min delta to trigger a report. |
| `reportingConfiguration(Integer cluster, Integer attrId[, Map])` | Hubitat-only. Reads the device's CURRENT reporting config (Read Reporting Configuration command, ZCL command `0x08`) — useful for diagnosing why a sensor isn't reporting. |
| `refreshData(String, String)` | Undocumented; signature suggests generic refresh dispatch. |
| `updateFirmware([Map])` | Hubitat-only. Issues OTA Image Notify (cluster `0x0019` command `0x00`). |

### Capability convenience builders

Each returns `List<String>` of pre-formatted `he raw`/ZCL commands ready to return from a driver `command`/`refresh` method.

- **Switch:** `on()`, `off()`, `onOffConfig()`, `onOffConfig(min, max)`, `onOffConfig(String attr, min, max)`, `onOffRefresh()`
- **Level:** `setLevel(level)`, `setLevel(level, rate)` (level 0–100, rate in tenths of a second, max 100 = 10s), `levelConfig()`, `levelRefresh()`
- **Color Temperature:** `setColorTemperature(K)` (Kelvin, ST notes "usually > 2700"), `colorTemperatureConfig()`, `colorTemperatureRefresh()`
- **Full color** (Hubitat-only, beyond what ST documented): `setColor(Map)`, `setColorXY(Map)`, `setHue(Integer)`, `setSaturation(Integer)`
- **Battery / Power:** `batteryConfig()`, `electricMeasurementPowerRefresh()`
- **Temperature:** `temperatureConfig()`, `temperatureConfig(min, max)`
- **IAS Zone:** `enrollResponse()` (cluster `0x0500` enrollment ack)

ST also exposed `simpleMeteringPowerConfig()` / `simpleMeteringPowerRefresh()` / `electricMeasurementPowerConfig()` / `hueSaturationRefresh()` and a `convertToInt(String, int radix)`. None of these is reachable on this firmware — Hubitat dropped them.

### Group commands (Hubitat-only)

Issue a multicast to a Zigbee group:

- `groupOn(Integer groupId)`, `groupOff(Integer groupId)`
- `setGroupLevel(Integer groupId, level[, rate])`
- `setGroupColorTemperature(Integer groupId, value[, rate, duration])`

### Utility

- `convertHexToInt(String)`, `convertToHexString(Integer value, Integer width)` (width is bytes×2 hex chars; `convertToHexString(10, 4) → "000A"`)
- `swapOctets(String)` (Hubitat-only naming; ST called the same thing `swapEndianHex`)
- `hex2String(String)`
- `clusterLookup(Object)` → `ZigbeeCluster` enum value (accepts int or name)

### Implementation/Groovy plumbing (skip)

`getCluster()`, `getDevice()/setDevice()`, `getMetaClass()/setMetaClass()`, `getProperty/setProperty`, `invokeMethod`, `__$swapInit`, `this$dist$*` — Groovy machinery, not API.

## `ZigbeeCluster` (the class the docs reference)

The Hubitat docs mention a `ZigbeeCluster` class. It's actually a **nested enum**: `com.hubitat.zigbee.Zigbee$ZigbeeCluster`. Returned by `zigbee.clusterLookup(int|String)` and the static `getCluster(Integer|String)` methods. Each enum value carries `clusterInt` (Integer), `clusterLabel` (String), and `clusterEnum` (String) accessors. The enum values mirror the integer constants above 1:1. Useful when you want a typed cluster reference instead of a bare int — but the bare-int constants are what every real driver uses.

## `ZoneStatus` (IAS Zone status bitfield)

FQN: `hubitat.zigbee.clusters.iaszone.ZoneStatus`. Wraps the 16-bit `ZoneStatus` attribute from cluster `0x0500`. Returned by `zigbee.parseZoneStatus(String)`.

Each property has both a direct-bit getter returning `0`/`1` and a boolean `isXxxSet()` query:

| Bit | Property | `getXxx()` returns | `isXxxSet()` true when |
|---|---|---|---|
| 0 | `alarm1` | 0/1 | opened / alarmed |
| 1 | `alarm2` | 0/1 | opened / alarmed |
| 2 | `tamper` | 0/1 | tampered |
| 3 | `battery` | 0/1 | low battery |
| 4 | `supervisionReports` | 0/1 | sensor reports supervision |
| 5 | `restoreReports` | 0/1 | sensor reports restore |
| 6 | `trouble` | 0/1 | trouble/failure |
| 7 | `ac` | 0/1 | AC/mains fault |
| 8 | `test` | 0/1 | in test mode |
| 9 | `batteryDefect` | 0/1 | defective battery detected |

Bit semantics preserved verbatim from the SmartThings docs.

## `DataType` (NOT reachable via reflection)

The ST docs documented a `physicalgraph.zigbee.zcl.DataType` class with the full ZCL type-code table and `getLength` / `isVariableLength` / `isDiscrete` / `pack` helpers. Hubitat's `zigbee` object exposes no equivalent through any visible return type — and `Class.forName("hubitat.zigbee.DataType")` is sandbox-blocked, so I can't probe whether the class exists under a different name.

If you need the constants, the practical options are:
- Hardcode the ZCL type codes you care about (`0x20` UINT8, `0x21` UINT16, `0x29` INT16, `0x30` ENUM8, `0x42` STRING_CHAR, etc.) — most drivers do this with named locals
- Use the SmartThings `DataType` table verbatim — the ZCL type codes haven't changed since at least 2018: see the captured table at the bottom of [zigbee-ref.html](https://web.archive.org/web/20200510195834/https://docs.smartthings.com/en/latest/ref-docs/zigbee-ref.html#datatype)
- Ask in the Hubitat developer forum whether there's a hidden `DataType` accessor — likely yes, given the helpers internally need it

## Conceptual notes worth keeping (from the ST primer)

- **Four main message types**: `command`, `readAttribute`, `writeAttribute`, `configureReporting`. Everything outside these (the old `raw …` family) is deprecated.
- **Endpoints**: *"basically ports. Different endpoints can support different clusters... a temperature sensor can have Temperature Measurement on endpoint 1 and OTA Boot loader on endpoint 2."*
- **Cluster taxonomy**: ZCL-defined / profile-specific (HA, ZSE) / manufacturer-specific. The cluster ID alone doesn't tell you which.
- **`additionalParams` rationale**: *"intended to support future params without affecting backward compatibility"* — explains the trailing `Map` instead of positional `mfgCode` and `destEndpoint`.
- **`zbjoin` schema** (from the primer, never re-documented by Hubitat): the `simple` string is space-separated `Endpoint, ProfileID, DeviceID, DeviceVersion, NumInClusters, [InClusters…], NumOutClusters, [OutClusters…]`. `application` = Basic cluster attribute `0x0001`; `manufacturer` = `0x0004`; `model` = `0x0005`.
- **Best practice the ST docs flag explicitly**: *"Do not use `sendEvent()` in command methods. Sending Events should be handled in the `parse` method."* Hubitat inherited this convention.

## What Hubitat added beyond SmartThings

Reachable now, not in the ST docs:
- Group commands (`groupOn`, `groupOff`, `setGroupLevel`, `setGroupColorTemperature`)
- Full color control: `setColor(Map)`, `setColorXY(Map)`, `setHue`, `setSaturation` (ST said *"Only color temperature can be parsed. Full color control is in the works."* — it shipped on Hubitat)
- `reportingConfiguration(...)` — read the device's current reporting config
- `updateFirmware([Map])` — OTA Image Notify
- `clusterLookup(...)` and the `ZigbeeCluster` enum
- Multi-attribute `readAttribute(cluster, List)`
- `parseZoneStatusChange(String)` (vs only `parseZoneStatus` in ST)
- Trailing `int delay` overloads for batching commands
- `temperatureConfig(min, max)`
- `String`-value overloads on `writeAttribute` (ST docs noted this gap)
- `swapOctets(String)` (renamed from ST's `swapEndianHex`)

## What ST had that Hubitat dropped (or hid)

- `convertToInt(String value, int radix)` — Hubitat has only `convertHexToInt(String)` (always base 16)
- `addBinding(...)` (use `command(BIND_MANAGEMENT_CLUSTER, …)` directly)
- `simpleMeteringPowerConfig()` / `simpleMeteringPowerRefresh()` / `electricMeasurementPowerConfig()` / `hueSaturationRefresh()`
- The full `DataType` helper class (see above)

## Gaps neither side ever documented

- `parseDescriptionAsMap` return-map field schema (reconstructed above from real DTHs)
- `getKnownDescription(String)`
- `parseZoneStatusChange(String)` return-map fields
- `refreshData(String, String)` purpose
- Whether `DataType` lives somewhere reachable from inside a driver

## Bonus: other classes discovered while walking

The introspection driver auto-follows `hubitat.*` / `com.hubitat.*` return types. The full method lists are in `hubitat-zigbee-introspection.md`; here's what's worth knowing they exist:

- **`hubitat.zigbee.SmartShield`** — what `zigbee.parse(String)` returns. Raw ZCL frame: `clusterId`, `command`, `profileId`, `sourceEndpoint`, `destinationEndpoint`, `data` (List), `manufacturerId`, `direction`, `messageType`, `options`, `isClusterSpecific`, `isManufacturerSpecific`. Constructed by the platform from a `catchall` description.
- **`com.hubitat.app.DeviceWrapper`** — what `zigbee.getDevice()` returns. Same wrapper drivers see as `device`. Of note: `getEndpointId()`, `getZigbeeId()`, `getDriverId()`, `eventsSince(Date)`, `statesSince(String, Date)`.
- **`com.hubitat.hub.domain.Hub`** — `getFirmwareVersionString()`, `getHardwareID()`, `getZigbeeEui()`, `getUptime()`. Reachable from `location.hubs[0]` (already noted in `hubitat-platform-notes.md`).
- **`com.hubitat.hub.domain.State`** — what `currentState(attr)` / `latestState(attr)` / `eventsSince(...)` return: `getStringValue`/`getNumberValue`/`getDateValue` accessors etc.
